1The Loop

Weaver's product is one flywheel with three verbs. Conjure is the core focus; the other two come nearly free because a widget is always its source.

ConjurePrompt your agent → a working widget on your desktop in under a minute. The primary authoring act; hand-writing code is the special case.
→
ShareSend the widget — which IS its source. What your friend reads is exactly what runs. Lineage travels in the manifest.
→
RemixTheir agent patches it to their taste — colors, size, data, anything. Every install is a potential fork. Every remix is a new share.

The two-minute demo

you, to your agent"make me a translucent now-playing widget with a seek bar, bottom left, dark, rounded — and a little audio visualizer that reacts to whatever's playing"

Thirty seconds later both exist on the desktop. That moment — conjuring — is what v1 is engineered around: the SDK, the agent skill, hot reload, and check-time validation all exist to make an agent one-shot that prompt reliably.

2A widget is one TypeScript component

One TSX module: config, state, and rendering in a single mental model that scales from a 10-line clock to a pomodoro app. (ADR 0001)

now-playing/widget.tsx 
import { widget, useProvider } from "@weaver/sdk";

export default widget({
  name: "Now Playing",
  size: [280, 96],
  anchor: { corner: "bottom-left", offset: [24, 64] },
  layer: "desktop",
  subscribe: ["media"],
  origins: [],                    // talks to no network — the media provider does the work
  capabilities: [],               // touches nothing on your system
}, () => {
  const media = useProvider("media");
  return (
    <row class="p-3 gap-3 items-center bg-[#101014]/85 rounded-2xl">
      <image src={media.art} class="w-11 h-11 rounded-lg" />
      <column class="min-w-0 grow gap-0.5">
        <text class="text-sm font-semibold truncate">{media.title}</text>
        <text class="text-xs text-[#9a9aa5]">{media.artist}</text>
        <slider value={media.position} max={media.duration}
                onChange={(p) => media.seek(p)} />
      </column>
    </row>
  );
});
  • Styling is free. Utility classes keep the Tailwind shape everyone already knows — including arbitrary values (bg-[#101014]/85) and any color. There is no theming system and no token law: the author's look is the intended vision, and making it yours is a remix, not a runtime override (ADR 0003). The built-in components (slider, button, charts…) have good defaults, so low-effort widgets still look decent.
  • State is normal React-shaped state. useState, useProvider, event handlers. Interactive widgets — seek bars, text inputs, settings — are the expected case, not an escape hatch.
  • Drawing is first-class. A <canvas> element with an immediate-mode draw API for visualizers, graphs, analog hands — data-space coordinates, any frame rate.
  • Check-time validation is mechanical, not aesthetic. Types, bindings vs subscriptions, undeclared origins, invalid capabilities — loud errors with fix-its. Taste is the author's (and the remixer's) domain.

4Fully alive, honestly billed

Any widget may animate at any rate. In exchange: idle-zero is a platform guarantee, and every widget's cost is always visible. (ADR 0005)

Now Playingby dara · remixed from @kae0.0% cpu · 12.1 MB
Spectrumby dara · conjured 07-124.2% cpu · 13.8 MB
Clockby @minimal-setups0.1% cpu · 11.4 MB
System Monitorby @rice-king · ⚠ reads files0.3% cpu · 12.9 MB
  • Idle-zero: a widget whose data isn't changing costs literally 0% CPU — event-driven repaints only; the platform has no polling render loop.
  • Visible bills: live CPU/RAM per widget in the manager, surfaceable as badges when sharing. "Make it under 3% CPU" is a spec an agent can be held to.
  • Crash isolation: one process per widget; one dies alone and the host restarts it.

5Sharing: the widget is the source

No sealed blobs, no author-choice opt-outs. What you read is what runs. (ADR 0004)

now-playing/
├── widget.tsx          — the whole widget: config, state, render
├── assets/             — optional images/fonts
└── weaver.lock         — provenance: author, version, remixed-from lineage

Install = Weaver builds the source locally (sub-second for a single TSX module). This is the decision that makes the Loop structurally cheap: sharing is sending a folder, remixing is your agent editing it, and the capability audit is reading the manifest fields at the top of the file you're about to run. Lineage fields exist from day one so a future gallery can treat "remixed from" as a first-class social credit.

6Platforms

Windows first (the customization culture lives there — and the windowing spike is already proven: transparent, desktop-layer, Win+D-surviving widgets at 11 MB / 0.6% CPU). macOS developed alongside, tested on real hardware. Linux acknowledged, very low priority. One platform seam internally; no OS-ism ever leaks into the SDK. (ADR 0006)

7Open engineering questions (in flight)

  • Substrate re-audit: TSX components mean an embedded JS engine per widget, and visualizers mean sustained 30–60fps — a different load profile than the compiled-markup/1Hz numbers our substrate gates measured. Spikes running now: QuickJS marginal footprint, and the high-FPS present path cost. Verdict lands in RESULTS.md.
  • High-FPS render path: the current layered-window present is proven at 1Hz; dirty-region/SIMD/GPU work is a headline workstream if the spike says 60fps is marginal.
  • TSX → native tree bridge: reconciler design (a Preact-class VDOM driving the native canvas tree over the component boundary).
